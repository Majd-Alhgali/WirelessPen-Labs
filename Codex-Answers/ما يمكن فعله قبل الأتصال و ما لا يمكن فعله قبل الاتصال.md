نعم. في مرحلة **قبل الاتصال** يجب أن نعيد تعريف الهدف بدقة: أنت لا تختبر الراوتر كجهاز IP، بل تختبر **طبقة Wi‑Fi نفسها** وما تكشفه قبل المصادقة.

**ما يمكن فعله قبل الاتصال**
1. **Passive Recon**
   - معرفة SSID/BSSID/channel/security.
   - تحديد هل الشبكة WPA2 فقط أم mixed `WPA/WPA2`.
   - معرفة هل التشفير `CCMP/AES` أم يوجد `TKIP`.
   - معرفة هل PMF/802.11w معلن كـ disabled/optional/required.
   - معرفة العملاء المتصلين من حزم الإدارة والـ data frames.
   - تقدير قوة الإشارة، ازدحام القناة، وسلوك الراوتر.

2. **WPS Assessment**
   - اختبار هل WPS مفعّل.
   - أنت قلت إنه غير مفعّل، إذًا هذا المسار يُغلق مبدئيًا.

3. **PMKID / Handshake Capture**
   - محاولة الحصول على PMKID بدون انتظار عميل.
   - أو التقاط handshake عند إعادة اتصال عميل.
   - هذا لا يكسر كلمة المرور وحده، لكنه يحدد هل لدينا مادة صالحة للاختبار offline لاحقًا.

4. **PMF / Deauth Resistance**
   - اختبار هل الشبكة تقاوم deauthentication.
   - إذا كانت PMF غير مفعلة أو اختيارية، يمكن فصل العملاء غالبًا.
   - ملفك السابق فيه آلاف Deauth frames، وهذا مؤشر أن الشبكة لا تمنع هذا السيناريو بشكل قوي.

5. **Client Behavior Analysis**
   - معرفة هل الأجهزة تكشف probe requests.
   - هل توجد أجهزة تعيد الاتصال بسرعة.
   - أي جهاز يعطي handshake أفضل.
   - هل توجد أجهزة قديمة تستخدم معدلات أو سلوك قديم.

6. **Evil Twin Simulation داخل اللاب**
   - مفيد تعليميًا لفهم سلوك العملاء عند وجود شبكة مقلدة.
   - لا نجمع كلمات مرور حقيقية ولا نبني صفحة تصيد حقيقية.
   - الهدف هنا: هل الأجهزة تحاول الاتصال؟ هل المستخدم يلاحظ؟ هل PMF/auto-join يقلل الخطر؟

**ما لا يمكن فعله قبل الاتصال**
- لا يمكن معرفة `<GATEWAY_IP>` بشكل موثوق.
- لا يمكن فحص UPnP.
- لا يمكن فحص لوحة الراوتر.
- لا يمكن معرفة firmware من الداخل.
- لا يمكن فحص أجهزة IoT عبر IP.
- لا يمكن اختبار default admin password.
- لا يمكن قراءة DHCP/ARP/IP traffic لأن البيانات مشفرة.

**الخطة العملية المقترحة**

1. **تجهيز أدوات Wi‑Fi**
   ```bash
   sudo apt install -y aircrack-ng hcxtools wireshark tshark macchanger iw
   ```

2. **تحديد الواجهة**
   ```bash
   iw dev
   ```

3. **وضع Monitor Mode**
   ```bash
   sudo airmon-ng check kill
   sudo airmon-ng start wlp4s0
   ```

4. **رصد الشبكة فقط**
   ```bash
   sudo airodump-ng wlan0mon
   ```

   نثبت:
   - BSSID: `80:AF:CA:CA:A3:D3`
   - Channel: `11`
   - ESSID: `Taim Starlink`
   - Security: هل هي `WPA2` فقط أم `WPA/WPA2`.

5. **التقاط مخصص للشبكة**
   ```bash
   sudo airodump-ng --bssid 80:AF:CA:CA:A3:D3 -c 11 -w taim_preconnect wlan0mon
   ```

   الهدف:
   - عملاء متصلون.
   - handshake إن حدث طبيعيًا.
   - beacon/probe/association frames.

6. **تحليل الالتقاط**
   ```bash
   aircrack-ng taim_preconnect-01.cap
   hcxpcapngtool --info=stdout taim_preconnect-01.cap
   ```

   نبحث عن:
   - PMKID.
   - EAPOL pairs.
   - PMF.
   - WPA/WPA2 mixed mode.
   - كثافة deauth/disassoc.

7. **اختبار WPS مرة واحدة**
   ```bash
   sudo wash -i wlan0mon
   ```

   إذا لم يظهر WPS أو ظهر locked/disabled، نغلق المسار.

8. **اختبار PMF بشكل محافظ**
   - نبدأ بتحليل الإعلان أولًا من beacon/RSN.
   - إذا كان PMF غير مطلوب، يمكن عمل اختبار deauth محدود جدًا داخل اللاب لتأكيد السلوك.
   - لا نستخدم deauth مستمر؛ فقط اختبار قصير لقياس هل يحصل reconnect وhandshake.

9. **تحليل العملاء**
   - نحدد أقوى عميل وأقربه.
   - نحدد من يعطي EAPOL كامل.
   - نحدد هل هناك أجهزة قديمة أو random MAC.
   - في ملفك السابق، العميل `3E:4F:AE:61:46:3F` كان مهمًا للـ EAPOL.

10. **تقرير الفجوات**
   نخرج بنتيجة مثل:
   - WPS: غير مفعّل.
   - PMF: غير مفعّل/اختياري/مطلوب.
   - WPA mode: WPA2 فقط أو mixed WPA/WPA2.
   - Cipher: AES فقط أو يوجد TKIP.
   - Handshake: موجود/غير موجود/جودته.
   - PMKID: موجود/غير موجود.
   - Deauth resistance: ضعيفة/متوسطة/قوية.
   - Client exposure: منخفض/متوسط/عالٍ.

**أهم نقطة في حالتك**
من المعطيات السابقة، الشبكة ظهرت أحيانًا كـ `WPA1 WPA2`، وهذا يستحق الفحص. إذا كان الراوتر يعمل بوضع mixed WPA/WPA2 بدل WPA2-AES فقط، فهذه فجوة إعدادات حتى لو لم تكسر كلمة المرور.

المسار الأكثر فائدة الآن ليس `<GATEWAY_IP>`. المسار الصحيح هو: **PMF + WPA mode + cipher + handshake quality + client behavior**. هذا هو قلب اختبار ما قبل الاتصال.
